# Day6・Day7 総合演習：作り込み・デプロイ・改善・発表

[TOC]

## この2日間の位置づけ

この総合演習では、アイデアの新しさよりも、**自分で仕組みを理解しながら、動くものを作り込むこと**を重視する。仕組みを理解して作れるようになることは、チームで働くときにも必要になるし、アイデアを自分の手で形にできる力につながる。完成度が高くなくても、自分でどう動いているかを説明できる状態を目指してほしい。

Day6 では、画面・UI など、ユーザーが直接触れる部分を中心に作り、1日目のうちにデプロイまで進める。まず URL で見られる状態、または動作確認できる状態にすることを優先する。

Day7 では、Day6 で作ったものをもとに改善する。データベース、認証、外部 API、micro:bit 連携などは必須ではないが、余裕があれば積極的に試してほしい。最後は全員が発表し、デモだけでなく「どういう仕組みで動いているか」も説明できるように準備する。

## 到達目標

**デザインする**

- 作るものを決め、画面構成と処理の流れを整理できる
- 仕様を言語化し、実装前に仕組みを整理できる

**実装する**

- 最小限でも動くアプリまたは作品を作れる
- 作ったものをデプロイし、他の人がアクセスできる状態にできる

**説明する**

- UI、入力、出力、データの流れを自分の言葉で説明できる
- 授業で扱った技術を作品の中で使い、その役割を説明できる
- 発表とレポートで、仕組みを具体的に説明できる

## 基本方針

**進め方**

- テーマ決定は短く済ませ、作り込みに時間を使う
- Day6 は UI とデプロイを優先する
- Day7 は改善を中心にする

**技術選択**

- データベース、API、認証、micro:bit は必須ではないが、できるだけ多くの技術を作品に取り入れる

**評価・発表**

- 使った技術の数だけでなく、その仕組みを理解して説明できるかを重視する
- 最後は全員が発表する

## Day6：UI 実装とデプロイ

### 作るもの

**まず目指すこと**

- ボタン、入力欄、一覧、表示エリアなど、操作できる UI がある
- デプロイ URL があり、他の人の端末からアクセスできる
- 自分の作品がどう動いているかを簡単に説明できる

**プラスアルファ**

- データベースと連携し、データを保存・取得する
- 外部 API を使ってデータを取得・表示する
- micro:bit と連携し、入力または出力として組み込む
- 認証を入れ、ログイン状態によって表示を変える

| 15分 |   個人   | 処理の流れを書く                                       |
| 10分 | グループ | 処理の流れを説明し、質問を受ける                       |
|  5分 |   説明   | 実装開始前チェック：最初に作るものを確認する           |

### このコマで決めること

**作品の概要**

- 何を作るか
- 今日デプロイする最小版は何か

**画面・操作**

- ユーザーは何を操作するか
- 画面または micro:bit には何が表示されるか

**データ・処理**

- 入力された値はどこで処理されるか
- どのデータを仮で持つか

> **ポイント：仕様書として書き出す**
>
> 作り始める前に、作るものの仕様を短い文章でまとめる。「ユーザーが何を入力し、何が返ってくるか」「画面にどう表示されるか」「どの処理がどこで動くか」を箇条書きにしたものでよい。
>
> これを書いておくことで、実装前に仕組みを頭で整理できる。また、AI にコードを書いてもらうときの指示にそのまま使えるほか、完成後に「仕様通りに動いているか」を確認する基準にもなる。

> **AI の使い方（1コマ目）**
>
> このコマは、ChatGPT や Claude などの **チャット AI** を中心に使う。作るものを整理したり、仕様を言語化したり、処理の流れを確認したりするのに向いている。コードを書き始めるより先に、AI と対話しながら設計を固める。

### このコマで進めること

**実装**

- UI を作り、操作できる状態にする
- 仮データでもよいので、入力と出力がつながるようにする
- micro:bit を使う場合は、UI の一部として組み込む

**デプロイ**

- GitHub にプッシュし、デプロイ URL を取得する
- 他の人の端末からアクセスできることを確認する

> **補足：デプロイの手順が分からない場合は早めに聞く**
>
> デプロイは時間がかかることがある。手順が分からない、エラーが出て止まっているといった場合は、後回しにせず早めに近くの人や講師に聞く。

**振り返り**

- 今作った部分がどう動くかをメモする
- できたこと・終わっていないこと・明日やることを整理する

> **AI の使い方（2コマ目）**
>
> このコマは、**AI コーディングエージェント**（Gemini CLI）を中心に使う。1コマ目で書いた仕様をそのまま指示として渡し、コードを生成・修正してもらう。エラーへの対応や修正も AI に依頼しながら進める。

> **ポイント：動いたらコミットする**
>
> 動く状態になったタイミングで `git commit` を残す。壊す前にセーブ、が基本。

### Day6 の振り返り

授業時間内に終わらなかった実装は、できる範囲で進めておく。

## Day7：改善・発展実装・全員発表

### 作るもの

**まず目指すこと**

- ボタン、入力欄、一覧、表示エリアなど、操作できる UI がある
- デプロイ URL があり、他の人の端末からアクセスできる
- 自分の作品がどう動いているかを簡単に説明できる

**プラスアルファ**

- データベースと連携し、データを保存・取得する
- 外部 API を使ってデータを取得・表示する
- micro:bit と連携し、入力または出力として組み込む
- 認証を入れ、ログイン状態によって表示を変える

### 最終成果物とレポート

提出するのは動く成果物とレポートの2つ。

**提出物**

- 成果物：デプロイ URL・動画・リポジトリリンク（自分以外の人に見せられる状態）
- レポート：作品名・概要・使い方・使った技術と仕組み（DB・API・micro:bit・認証など使ったものは何のために使いどう動いているか）・頑張った点・改善したい点

AI を使って構成を整えてよいが、書いた内容は自分で理解している必要がある。全部埋まっていなくてもよいので、今日中に一度提出する。

> **レポート例**
>
> **作品名**：タスク管理アプリ
>
> **概要**：ToDoリストをブラウザで管理できる Web アプリ。ログインすると自分のタスクだけ表示される。
>
> **使い方**：ログイン後、入力欄にタスクを入力して追加ボタンを押す。完了したらチェックを入れ、削除ボタンで消せる。
>
> **使った技術と仕組み**：
>
> - **データベース**：タスクの保存に使用。追加ボタンを押すと fetch で POST リクエストをサーバーに送り、データベースに保存する。ページ読み込み時に GET で取得して一覧表示する。
> - **認証**：ログインした人だけ自分のタスクを見られるようにするために使用。JWT でトークンを発行し、ローカルストレージで管理している。ログアウトするとトークンを削除し、タスクにアクセスできなくなる。
>
> **頑張った点**：タスクの完了状態をページ再読み込みなしで更新できるようにした。
>
> **うまくいかなかった点**：デプロイ後に環境変数の設定でエラーが出て時間がかかった。次はデプロイ手順をメモに残しておきたい。

**説明・改善**

- レビューを受けて改善した跡がある
- 発表とレポートで仕組みを具体的に説明できている
