# Day6・Day7 総合演習
## 作り込み・デプロイ・改善・発表

---

## この2日間の位置づけ

アイデアの新しさよりも、**自分で仕組みを理解しながら、動くものを作り込むこと**を重視する

- 仕組みを理解して作れるようになる → チームで働く力・アイデアを形にする力につながる
- 完成度が高くなくても、**自分でどう動いているかを説明できる状態**を目指す

**Day6**：画面・UI を中心に作り、1日目のうちにデプロイまで進める

**Day7**：Day6 で作ったものをもとに改善する。最後は全員が発表する

> データベース、認証、外部 API、micro:bit 連携は必須ではないが、余裕があれば積極的に試してほしい

---

## 到達目標

**デザインする**

- 作るものを決め、画面構成と処理の流れを整理できる
- 仕様を言語化し、実装前に仕組みを整理できる

**実装する**

- 最小限でも動くアプリまたは作品を作れる
- 作ったものをデプロイし、他の人がアクセスできる状態にできる

**説明する**

- UI、入力、出力、データの流れを自分の言葉で説明できる
- 授業で扱った技術を作品の中で使い、その役割を説明できる
- 発表とレポートで、仕組みを具体的に説明できる

---

## 基本方針

**進め方**

- テーマ決定は短く済ませ、作り込みに時間を使う
- Day6 は UI とデプロイを優先する
- Day7 は改善を中心にする

**技術選択**

- データベース、API、認証、micro:bit は必須ではないが、できるだけ多くの技術を作品に取り入れる

**評価・発表**

- 使った技術の数だけでなく、**その仕組みを理解して説明できるか**を重視する
- 最後は全員が発表する

---

## Day6・Day7 で目指すこと

**まず目指すこと**

- ボタン、入力欄、一覧、表示エリアなど、操作できる UI がある
- デプロイ URL があり、他の人の端末からアクセスできる
- 自分の作品がどう動いているかを簡単に説明できる

**プラスアルファ**

- データベースと連携し、データを保存・取得する
- 外部 API を使ってデータを取得・表示する
- micro:bit と連携し、入力または出力として組み込む
- 認証を入れ、ログイン状態によって表示を変える

# Day6：UI 実装とデプロイ

---

## 1コマ目：制作計画・UI設計・仕組みの整理（100分）

| 時間 | 種別 | 内容 |
| ---: | :---: | --- |
| 10分 | 説明 | 導入：2日間のゴール、作り込み重視、1日目デプロイの確認 |
| 10分 | 個人 | 作るものを1つに決める |
| 15分 | 個人 | 今日作る最小機能を決める |
| 15分 | グループ | 作るもの、操作方法、使う技術を説明し、質問を受ける |
| 20分 | 個人 | 画面、UI、micro:bit の使い方を決める |
| 15分 | 個人 | 処理の流れを書く |
| 10分 | グループ | 処理の流れを説明し、質問を受ける |
| 5分 | 説明 | 実装開始前チェック：最初に作るものを確認する |

---

## 1コマ目で決めること

**作品の概要**

- 何を作るか
- 今日デプロイする最小版は何か

**画面・操作**

- ユーザーは何を操作するか
- 画面または micro:bit には何が表示されるか

**データ・処理**

- 入力された値はどこで処理されるか
- どのデータを仮で持つか

---

## ポイント：仕様書として書き出す

作り始める前に、作るものの仕様を短い文章でまとめる

> 「ユーザーが何を入力し、何が返ってくるか」「画面にどう表示されるか」「どの処理がどこで動くか」を箇条書きにしたものでよい

これを書いておくことで：

- 実装前に仕組みを頭で整理できる
- AI にコードを書いてもらうときの指示にそのまま使える
- 完成後に「仕様通りに動いているか」を確認する基準になる

---

## AI の使い方（1コマ目）

このコマは、**チャット AI**（ChatGPT や Claude）を中心に使う

- 作るものを整理したり、仕様を言語化したりするのに向いている
- 処理の流れを確認したり、設計を固めたりするために使う

コードを書き始めるより先に、**AI と対話しながら設計を固める**

## 2コマ目：UI実装・修正・デプロイ（100分）

| 時間 | 種別 | 内容 |
| ---: | :---: | --- |
| 35分 | 個人 | UI 実装、micro:bit 連携、仮データ実装 |
| 15分 | グループ | グループ内デモ：ローカルまたは micro:bit 込みで見せる |
| 35分 | 個人 | デモで出た質問をもとに修正し、デプロイする |
| 10分 | グループ | デプロイ URL 確認：グループ内でアクセスできるか確認する |
| 5分 | 個人 | 振り返り：できたこと、終わっていないこと、明日やること |

---

## 2コマ目で進めること

**実装**

- UI を作り、操作できる状態にする
- 仮データでもよいので、入力と出力がつながるようにする
- micro:bit を使う場合は、UI の一部として組み込む

**デプロイ**

- GitHub にプッシュし、デプロイ URL を取得する
- 他の人の端末からアクセスできることを確認する

**振り返り**

- 今作った部分がどう動くかをメモする
- できたこと・終わっていないこと・明日やることを整理する

---

## AI の使い方（2コマ目）

このコマは、**AI コーディングエージェント**（Gemini CLI）を中心に使う

- 1コマ目で書いた仕様をそのまま指示として渡し、コードを生成・修正してもらう
- エラーへの対応や修正も AI に依頼しながら進める

> **ポイント：動いたらコミットする**
>
> 動く状態になったタイミングで `git commit` を残す。壊す前にセーブ、が基本。

---

## デプロイが分からない場合は早めに聞く

デプロイは時間がかかることがある

- 手順が分からない
- エラーが出て止まっている

→ **後回しにせず早めに近くの人や講師に聞く**

---

## Day6 の振り返り

Day7 の最初にグループで成果物を見せ合う。**デプロイまで終わらせておく**。

授業時間内に終わらなかった実装は、できる範囲で進めておく。

次回の作業にすぐ入れるように整理しておく：

**できたこと**：実装した機能、デプロイの状態

**うまくいかなかったこと**：動いていないところ、どう動いているか分からないところ

**次にやること**：修正・追加したいこと、発表で見せたいところ

## 発表フォーマット

1人 **4分** を目安に話す

1. 何を作ったか・誰が使うか・**デモ**
2. **どういう仕組みで動いているか**
3. 頑張った点・改善したい点

---

## 最終成果物とレポート

提出するのは**動く成果物**と**レポート**の2つ

**提出物**

- 成果物：デプロイ URL・動画・リポジトリリンク（自分以外の人に見せられる状態）
- レポート：作品名・概要・使い方・使った技術と仕組み・頑張った点・改善したい点

> AI を使って構成を整えてよいが、書いた内容は自分で理解している必要がある

---

## レポート例

**作品名**：タスク管理アプリ

**概要**：ToDoリストをブラウザで管理できる Web アプリ。ログインすると自分のタスクだけ表示される。

**使った技術と仕組み**：

- **データベース**：タスクの保存に使用。fetch で POST → サーバーが DB に保存。GET で取得して一覧表示。
- **認証**：JWT でトークンを発行しローカルストレージで管理。ログアウトするとトークンを削除。

**頑張った点**：タスクの完了状態をページ再読み込みなしで更新できるようにした。
