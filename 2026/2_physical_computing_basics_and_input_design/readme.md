# Day2 フィジカルコンピューティング：基礎・入力・出力

[TOC]

## この日の位置づけ

Day1でAI時代のものづくりの全体像を掴んだ次のステップとして、実際に手を動かしてフィジカルコンピューティングの基礎を体験する。micro:bit と MakeCode を使い、「デジタルが物理世界とつながる」感覚をつかむ。

1コマ目でDay1の質問への回答、概念、micro:bit / MakeCode の基本、入力と出力の考え方を固める。2コマ目で（私物）デバイスの調査、デモ、AIコード生成、個別課題・チャレンジに取り組む。1日で入力から出力まで触れることで、次回（Day3）の応用制作につながるイメージを持ちやすくする。

授業はハイブリッド形式で実施する。現地参加者は実機の micro:bit を使い、リモート参加者は MakeCode のシミュレーターで同じ実習を行う。

## 到達目標

- フィジカルコンピューティングの概念と代表的な活用事例を説明できる
- micro:bit と MakeCode の基本操作を習得し、自分でプログラムを書いて実行できる
- AI（Copilot・ChatGPT等）を使ってMakeCodeのJavaScriptコードを生成・修正できる
- AIが生成したコードが何をしているか、自分の言葉で説明できる
- micro:bit のボタン・加速度センサー・温度センサーのデータを取得できる
- LEDマトリクスとスピーカーを使った出力プログラムを組める
- AIがフィジカルコンピューティングの表現をどう拡張しているか、自分の言葉で話せる

### 1. Day1の質問への回答【講義】

Day1のワークショップで出た質問・不安に対して、Day2の入口で短く答える。

主な回答：

- **AIの情報をどこまで信頼していいのか**：基本は「疑いながらやりとりする」。ただし、スピードのためにある程度でいったん信じて試す。違和感を感じたら、その違和感をAI自身に伝えて「どこが怪しいか」「何を確認すべきか」を再度深掘りする。
- **上手な指示の出し方は何か**：AIに「どう頼めばいいか」自体を聞く。一段高い視点に上がって、「目的を整理して」「足りない条件を質問して」「プロンプトを改善して」と頼ってよい。これはAIを使うときのメタ思考でもある。
- **ChatGPTはなぜ肯定的に返すのか**：そう返すように学習されている面がある。ユーザーは肯定的で感じのよい返答を求めている、という前提で振る舞いやすい。だから、あえて「反対意見を出して」「リスクを指摘して」「間違いがないか疑って」と頼む。
- **どんなアプリや作品が作れるのか**：生活の困りごと、学校の不便、遊び、表現を小さな入出力に分解すれば、Day6-7の制作テーマにつながる。
- **先生方が今気になっているAI技術は何か**：センサーで得た状況をAIが認識・判断し、ロボットやアプリの次の動作につなげるPhysical AIの流れ。

具体的な不安への回答：

- **「コードの意味が分からない」**：今日は全部を暗記しない。入力、判断、出力の3つに分けて、どの行が何を担当しているかを見る。
- **「うまく動かないときに原因追求が難しそう」**：エラーは失敗ではなく、観察する材料として扱う。好奇心と探究心を大事にして、「何が起きているのか」「どこまで分かっているのか」をAIにも聞きながら切り分ける。
- **「ひとりで何かできるようになるのか」**：最初はAI、友人、教員に頼ってよい。大事なのは、最後に何を変えたら動いたかを自分で説明できること。
- **「物理・理工系っぽくなりすぎないか」**：そこは授業側でも気をつける。回路は深掘りせず、さわりだけ扱う。基本は、つなげば動くデバイスがたくさんあるので、必要なものをAIや資料で一緒に調べながら使う。もっと深掘りしたい人は質問してよい。

### 2. フィジカルコンピューティングとは【講義】

デジタルと物理世界をつなぐとはどういうことか、事例と合わせて説明する。

代表的な事例：

- インタラクティブアート（teamLab, Zimoun など）
- スマート家電・IoTデバイス
- 体験型インスタレーション
- ロボット制御（センサーで障害物を検知して動きを変える、腕の動きをモーターに伝えるなど）
- 自動車・モビリティ（カメラ・LiDARで周囲を認識して走行・停止を判断する自動運転、衝突検知センサーなど）
- AIとセンサーを組み合わせた作品（音声認識 + LED、顔認識 + 物理フィードバックなど）

AI時代のフィジカルコンピューティング：

- センサーデータをAIモデルに渡すことで、より豊かなインタラクションが可能に
- センシングした状況をAIが認識・判断し、次の動作につなげるシステムが増えている
- コード生成AIを使えば、プログラミング経験が少なくてもプロトタイプを作れる
- 表現のアイデアを先に決め、実装はAIと協力する開発スタイル

フィジカルAI（Physical AI）の広がり：

- 近年、AIが「デジタルの世界だけ」から「物理世界に作用する」方向へ進化している
- ロボットや自律走行車、ドローンなど、センサーで環境を認識してリアルタイムに動作を決定するシステム
- NVIDIAが提唱する「Physical AI」の概念：AIが視覚・触覚・音など複数のセンサー情報を統合して判断する
- 身近な例：スマートスピーカー（音声 → AIが判断 → 照明や家電を操作）、自動運転（カメラ・LiDAR → AIが判断 → 操舵・加速・停車）
- 最近の例：中国・北京のヒューマノイドロボットハーフマラソン。ロボットが現実のコースで、バランス、経路、状況変化に対応しながら走る事例として紹介する。
  - 2026年の報道例: https://www.theguardian.com/sport/2026/apr/19/humanoid-robots-race-beijing-half-marathon
  - 2025年の初回報道例: https://www.reutersconnect.com/item/china-pits-humanoid-robots-against-humans-in-half-marathon/dGFnOnJldXRlcnMuY29tLDIwMjU6bmV3c21sX0xWQTAwMTc2OTAxODA0MjAyNVJQMQ
- 今日やる micro:bit の入力→処理→出力の流れは、これらの大規模なシステムと根っこは同じ

### 3. micro:bit と MakeCode の基本・チュートリアル【講義＋実習】

micro:bit の各部位を確認する：

- 5×5 LEDマトリクス
- ボタン A / B
- 加速度センサー、温度センサー、光センサー、コンパス
- スピーカー（v2）
- エッジコネクタ（外部接続ピン）

MakeCode の基本操作を確認する：

- ブラウザで開く（makecode.microbit.org）
- ブロックとJavaScriptの切り替え
- シミュレーターで動作確認（リモート参加者はここで実習を完結できる）
- micro:bit への書き込み手順（USBドライブとして転送）※現地参加者のみ

**実習**：MakeCodeの公式チュートリアルを、教員と一緒に全員で同じ手順で試す。

**課題**：実習で試した内容をもとに、各自で記録を残す。

- チュートリアルを進めながら「このブロックは何をしているか」を確認していく
- チュートリアルで作った画面のスクリーンショットを1枚残す
- どのチュートリアルを試し、どこを理解したかを1〜2文で書く
  例：「ボタンを押すと表示が変わるチュートリアルを試した。`on button pressed` で入力を受け取り、LED表示を変える流れがわかった」
- 評価の基準：動いた結果だけでなく、入力と出力の関係を自分の言葉で説明できているかを見る。

オプション課題（余裕がある人向け）：

- チュートリアルで作ったプログラムを自分なりにアレンジしてみる（表示するアイコンを変える、タイミングを変えるなど）

### 5. 出力の確認【講義】

LEDマトリクスとスピーカーでできることを説明する。

LEDマトリクスでできること：

- アイコン・文字のスクロール表示
- 座標指定で個別LEDを点灯（`led.plot(x, y)`）
- 明るさの変化・アニメーション

スピーカーでできること：

- 音階の再生（`music.playTone`）
- ビルトインメロディの再生
- センサーの値に応じて音を変える

### 9. AIを使ったコード生成とコード読解【講義＋実習】

チュートリアルで動かしたプログラムをJavaScriptで見てみる。ブロックとJavaScriptが対応していることを確認したうえで、AIにコードを生成させ、動かし、読み解くところまで一緒に行う。

AIにコードを生成させるときのポイント：

- プロンプトに「MakeCode JavaScript で〜」と指定する
- 出力を貼り付けて動作確認する
- うまく動かないときはAIとデバッグ会話をする

あわせて、AIを「答えを出す道具」だけでなく「発想を広げる相手」として使うことも意識する。

- 好奇心と探究心があるほど、AIから引き出せるものが増える
- AIで学びを加速するには、質問、実験、確認を何度も往復する
- いきなり答えを1つ出してもらうだけでなく、複数案を出させて比べる
- 自分の興味ややってみたい体験を伝えて、案を具体化する
- 生成されたコードをそのまま使うのではなく、何をしているか説明できるようにする
- 最後は、理解・納得・感受性がボトルネックになる

**実習**：以下のプロンプトでAIにコードを生成してもらい、教員と一緒に動かしてみる。

- 「MakeCode JavaScript で、ボタンAを押すたびにカウントアップしてLEDに表示するコードを書いて」
- 「MakeCode JavaScript で、micro:bit を振るとランダムなアイコンを表示するコードを書いて」
- 「MakeCode JavaScript で、LEDにアニメーションを表示するコードを書いて」
- 「MakeCode JavaScript で、温度が高いほど音が高くなるコードを書いて」

動かした後に、以下を確認する。

- 入力はどこか
- 出力はどこか
- 変数は何を記憶しているか
- `if` は何を判断しているか
- `forever` やループは何を繰り返しているか

### 10. 個別課題：作る・読む・提出する【課題】

**課題**：各自でAIを使ってプログラムを1つ作り、動かし、コードを読んで、文章で提出する。

- AIにMakeCode JavaScriptのコードを作ってもらう
- MakeCodeに貼り付けて動かす
- 動いたコードを読んで、入力・出力・変数・条件分岐・ループを確認する
- わからない行はAIに「この行は何をしているの？」と聞いて理解する（「なんとなく動いた」ではなく「なぜ動くか話せる」状態を目指す）
- 作成したプログラムの画面キャプチャを1〜2枚残す
- 実機で試した人は動作している様子を写真で1枚残す。リモート参加者はシミュレーター画面のキャプチャでよい
- 指定フォーマットに沿って文章で提出する
- 評価の基準：AIの出力をそのまま貼るだけでなく、動作確認とコード読解ができているかを見る。

提出フォーマット：

```text
作ったものの名前：

使った入力：

起きる出力：

AIに最初に投げたプロンプト：

追加でAIに聞いたこと：

コード読解：

動かして分かったこと・直したこと：
```

チャレンジ（余裕がある人向け）：

- デモのゲームを自分なりに改造する
- （私物）デバイスを使った作品案をAIに相談して、MakeCodeで試せる形まで具体化する
- 複数のセンサーや出力を組み合わせる（例：ボタンで温度計のオン/オフを切り替える）
