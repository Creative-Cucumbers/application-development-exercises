## この日の位置づけ

Day2・Day3 でフィジカルコンピューティング（センサー入力・出力・組み合わせ）を体験した。Day4 からはアプリ開発の領域に入る。担当も伊藤から小島に交代する。

1コマ目では、開発環境（VSCode / Git / Gemini CLI）のセットアップを確認したうえで、Gemini CLI を使ったバイブコーディングを体験する。2年生で学んだバイブコーディングをコーディングエージェントの文脈で再整理し、シンプルな投票アプリを作る。

2コマ目では、「作ったアプリのデータがリロードで消える」という体験から「なぜデータベースが必要か」を自然に導入する。構造化データと非構造化データを対比し、AIが非構造化データの扱いを変えたことを理解する。最後に SQLite を使ってデータを実際に操作し、Day5 のバックエンド接続への布石とする。

授業はハイブリッド形式で実施。リモート参加者もブラウザ上で同じ実習に取り組める。

## 到達目標

- VSCode・Git・Gemini CLI のセットアップができ、基本的な使い方を説明できる
- Gemini CLI に指示を出し、HTML/JS のシンプルなフロントエンドを作れる
- 「データがメモリにしか存在しない」状態と「データが永続化された」状態の違いを説明できる
- 構造化データと非構造化データの違いを説明できる
- AI によって非構造化データの扱い方と、アプリとの連携の選択肢が広がったことを説明できる
- SQLite でテーブルを作成し、INSERT・SELECT でデータを操作できる

### 3. バイブコーディング：投票アプリを作る【実習】

Gemini CLI に指示して、シンプルな投票アプリ（HTML/CSS/JS）を作る。

#### アプリの仕様

- 質問と選択肢（例：「今日のランチは何を食べたい？」/ ラーメン・寿司・カレー・サラダ）
- 各選択肢にボタンがあり、クリックするとカウントアップ
- 現在の集計結果をリアルタイムに表示
- フレームワークは使わず、純粋な HTML / CSS / JS で作る

#### Gemini CLI への最初のプロンプト例

```
シンプルな投票アプリを HTML/CSS/JS で作って。
機能：
- 質問：「今日のランチは何を食べたい？」
- 選択肢：ラーメン / 寿司 / カレー / サラダ
- 各選択肢にボタンがあり、クリックするとカウントアップ
- 現在の集計結果を表示する
フレームワークは使わず、1つの HTML ファイルにまとめて。
```

#### 制作フロー

1. Git でプロジェクトフォルダを初期化する（`git init`）
2. Gemini CLI にプロンプトを投げてコードを生成させる
3. VSCode でコードを開き、ブラウザで動作確認する
4. 気に入らない部分を Gemini CLI に修正させる
5. 動いたら `git add` → `git commit` で保存する

#### この時点での成果物

- ブラウザで動く投票アプリ（`index.html`）
- Git で管理されたプロジェクトフォルダ

### 4. 「消えた」問題の確認【実習＋問いかけ】

全員でページをリロードして、カウントが 0 に戻ることを確認する。

問いかけ：

- 「データはどこに行った？」
- 「ブラウザを閉じたら、投票結果はどうなる？」
- 「複数人が別々のブラウザで投票したら、結果は合計されるか？」

確認できること：

- データは今、ブラウザのメモリ（JavaScript の変数）にしか存在しない
- リロードや閉じると消える
- 別のブラウザとは共有されない

→ 「このデータを残すには、どこかに保存する必要がある」という問題提起で 2 コマ目へつなぐ。

### 5. なぜデータが重要か【講義＋問いかけ】

1 コマ目で体験した「消えた」から始めて、データとアプリの関係を整理する。

「消えた」の原因：

- JavaScript の変数はメモリ上にある
- ページを閉じるとメモリが解放される
- 「永続化」＝データをどこかに書き込んで残すこと

身近なアプリのデータを考える：

- SNS の投稿はどこに？
- EC サイトのカート情報は？
- LINE のトーク履歴は？

アプリにとってのデータ：

- ユーザーが入力したもの（投票・コメント・購入履歴）
- アプリが計算したもの（集計・推薦・スコア）
- 外の世界から来るもの（センサー値・API レスポンス・画像）

---

### 6. データの種類：構造化・非構造化・AIで変わったこと【講義＋実習】

#### 構造化データ（10分）

整理された形式で保存されているデータ。

例：

- 投票データ：`{ "ラーメン": 5, "寿司": 3, "カレー": 7, "サラダ": 2 }`
- 商品一覧：id / 商品名 / 価格 / 在庫数
- 気温ログ：日時 / 温度 / 場所

JSON を手で書いてみる実習：

- 投票アプリのデータを JSON で表現する
- Gemini CLI に「この投票アプリのデータを JSON 形式で表現して」と頼む
- キーと値の構造を確認する

#### 非構造化データ（10分）

決まった形式を持たないデータ。

例：

- 画像（JPEG / PNG）
- 音声・動画
- 自由記述テキスト（レビュー・コメント）
- センサーのストリームデータ（Day2・Day3 で触った micro:bit の加速度や温度も、そのままでは非構造化に近い）

従来のアプリとの連携：

- 画像をアプリで使うには「どの画像か」というパスを保存するのみ
- 中身（写真に何が映っているか）はアプリからは見えない
- 自由記述テキストを検索・分類するのが難しかった

#### AI によって何が変わったか（10分）

AI がデータ変換の担い手になった。

| 非構造化データ | AI による変換 | アプリとの連携 |
| --- | --- | --- |
| 画像 | 画像の説明文を生成 | テキストとして DB に保存・検索できる |
| 音声 | 文字起こし | テキストとして処理できる |
| 自由記述テキスト | 感情分析・カテゴリ分類 | 構造化して集計・フィルタできる |
| センサーストリーム | 状態の推定・異常検知 | 意味のある値として保存できる |

変化のポイント：

- 非構造化データを「AI API に投げると JSON が返ってくる」
- その JSON をそのまま DB に保存・表示できる
- アプリとの連携の選択肢が大幅に増えた

Day3 との接続：

- micro:bit のセンサーデータ（傾き・温度・音）も非構造化データ
- AI に投げると「激しく動いている」「静止している」などの状態に変換できる
- Day5・Day6 でフィジカルとアプリをつなぐときにこの考え方が活きる

### 7. データベースとは何か【講義】

「JSON ファイルに保存すればいいのでは？」という問いから始める。

JSON ファイル保存の限界：

- 複数人が同時に書き込むと壊れる
- データが増えると検索が遅くなる
- 「ラーメンに投票した人だけ抽出」のような操作が難しい

データベースが解決すること：

- 同時アクセスの制御
- 効率的な検索・集計
- データの整合性の保証

種類：

| 種類 | 特徴 | 例 |
| --- | --- | --- |
| RDB（リレーショナル） | 表形式・SQL で操作 | SQLite / PostgreSQL / MySQL |
| NoSQL | 柔軟な形式（JSON に近い）| MongoDB / Firebase |

SQLite の特徴：

- データベース全体が 1 つの `.db` ファイル
- サーバー不要、インストール不要（ブラウザ版で今日は使う）
- 実体が見えるので、「データベースとは何か」を理解しやすい
- Day5 では Node.js サーバーと組み合わせて外部に公開する

### 8. SQLite 実践【実習】

ブラウザで動く SQLite ツール（https://sqliteonline.com）を使い、投票アプリのデータを実際に操作する。Windows / Mac どちらでも同じ手順で動く。

#### 手順

**テーブル作成**：

```sql
CREATE TABLE votes (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  option TEXT NOT NULL,
  voted_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

**データ挿入**：

```sql
INSERT INTO votes (option) VALUES ('ラーメン');
INSERT INTO votes (option) VALUES ('寿司');
INSERT INTO votes (option) VALUES ('ラーメン');
INSERT INTO votes (option) VALUES ('カレー');
INSERT INTO votes (option) VALUES ('ラーメン');
```

**集計クエリ**：

```sql
SELECT option, COUNT(*) AS count
FROM votes
GROUP BY option
ORDER BY count DESC;
```

#### Gemini CLI を使った SQL 生成

ブラウザの Gemini CLI に頼んでみる：

- 「votes テーブルを作る SQL を書いて。カラムは id（自動採番）、option（選択肢名）、voted_at（投票日時）」
- 「option ごとの集計を出す SQL を書いて」
- 「直近 10 件の投票を新しい順に取得する SQL を書いて」

確認ポイント：

- テーブルの構造（スキーマ）と、1 コマ目の投票アプリのデータが対応していることを確認する
- SQL の各キーワード（SELECT / FROM / WHERE / GROUP BY）が何をしているか説明できるようにする

提出課題：

- 作成した votes テーブルの画面キャプチャを 1 枚
- 集計クエリの結果画面キャプチャを 1 枚
- 「データベースを使うと何が解決されるか」を 1〜2 文で書く
