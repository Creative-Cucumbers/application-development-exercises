## この日の位置づけ

初回講義では、授業全体の目的と進め方を確認したうえで、この授業で扱う「本格的なWebアプリ開発技術」と「フィジカルコンピューティング」の全体像をつかむ。

### 1-3. この授業のねらい（20分）

#### 前提知識の確認

**皆さんが2年生で学んだこと**
- AIとは何か、生成AIの基礎
- バイブコーディングでのアプリ作成
- フロントエンド開発の基礎

**3年生ではさらなるレベルアップを目指す**
- バックエンド技術の理解と実装
- データベース設計と操作
- ユーザー認証・権限管理
- フィジカルコンピューティングとの統合

#### 授業の3つの柱

**1. 本格的なアプリ開発技術の習得**
- **バックエンド**: Node.js、Python等でAPI開発
- **データベース**: Firebase、Supabase等でデータ管理
- **認証**: ユーザー登録・ログイン・権限管理の実装
- **具体例**: 2年生で作ったフロントエンドに、本格的なバックエンドとDBを組み合わせる

**2. フィジカルコンピューティング**
- センサーによる物理情報の取得
- アクチュエータによる物理世界への出力
- デジタルと物理をつなぐインタラクション設計
- **具体例**: マイコンとセンサーで環境データを取得し、Webアプリで可視化・制御

**3. 総合演習による統合**
- フィジカル + フロントエンド + バックエンド + データベースを統合
- 自分でテーマを設定し、企画・設計・実装・発表
- レビューとフィードバックによる改善
- **具体例**: IoTデバイスからデータを取得し、Webアプリで可視化・分析し、デバイスを遠隔制御するシステム

#### なぜ今、バックエンド・データベース・フィジカルなのか

**バックエンド・データベースの重要性**
- 2年生で学んだフロントエンドだけでは、本格的なアプリは作れない
- データの永続化、ユーザー管理、セキュリティは必須技術
- **例**: SNSアプリを作るには、投稿データの保存、ユーザー認証、APIが必要

**フィジカルコンピューティングの重要性**
- デジタル体験を物理世界に拡張することで、新しい価値を生み出せる
- スマートホーム、ウェアラブル、ロボティクスなど成長市場
- 「画面の中だけ」ではない、五感に訴える体験設計

**統合による新しい可能性**
- IoT、スマートシティ、インダストリー4.0など、統合が当たり前の時代
- 物理世界のデータを収集 → データベースに保存 → 分析 → 物理世界に働きかける
- **例**: 温度センサーでデータ取得 → DBに蓄積 → グラフ表示 → エアコンを自動制御

#### この授業を通じて目指す姿

**2年生からのレベルアップ**
- フロントエンドだけ → **フルスタック開発**（フロント + バック + DB）
- バイブコーディング → **コードベースの開発**（Node.js、Python等）
- 画面の中だけ → **物理世界と連携**（IoT、フィジカルコンピューティング）

**技術を「使いこなせる」人になる**
- バックエンド、データベース、認証の仕組みを理解し実装できる
- フィジカルデバイスとWebアプリを連携させられる
- AIを活用しつつ、本質を理解する

**課題を「解決できる」人になる**
- 技術ありきではなく、課題起点で考える
- 適切な技術を選択し、実装して検証し、改善するサイクルを回す

**価値を「届けられる」人になる**
- 作って終わりではなく、ユーザーに届けることを意識
- プレゼンテーションやデモで伝える力を養う

### 1-4. AI 時代の表現者・実装者ロードマップ：授業の位置づけ（15分）

このセクションでは、AI時代に求められる能力の全体像と、この授業で身につける5つの段階を説明します。

#### なぜ「表現者・実装者」ロードマップなのか

**AI時代の境界の崩壊**
- 従来：デザイナー、エンジニア、クリエイターが明確に分かれていた
- AI時代：表現と実装の境界が曖昧になり、両方にまたがる力が必要
- **重要**: 職業を早期に固定するのではなく、可動域を広げることが大切

**2年生からのレベルアップ**
- 2年生：AIの基礎、バイブコーディング、フロントエンド
- 3年生：**表現と実装を横断する力**を養う
  - バックエンド・データベース・認証技術
  - フィジカルコンピューティング
  - AI活用しながら構想を形にする力

#### 生成AI・ノーコードがものづくりをどう変えたか

| 工程 | 従来の進め方                       | AI / ノーコード活用で起きた変化      | この授業で重視すること                     |
| ---- | ---------------------------------- | ------------------------------------ | ------------------------------------------ |
| 発想 | 個人で時間をかけて調査・構想する   | 短時間で多数の案を出し、比較できる   | 良い問いを立て、案を選び直す力             |
| 試作 | 実装前の準備に時間がかかる         | 画面案、コード、回路案をすぐに試せる | まず小さく試し、仮説を検証する力           |
| 実装 | 文法や手順の習得が先に必要         | 動くものの初速は上がる               | 動いた理由を理解し、保守できる力           |
| 検証 | 人手での確認に頼りやすい           | AIに観点出しやデバッグ補助を頼める   | 出力を鵜呑みにせず、妥当性を検証する力     |
| 発表 | 制作意図の整理が後回しになりやすい | 資料や説明文の下書きを素早く作れる   | 価値、根拠、改善点を自分の言葉で説明する力 |

**重要**
- (現在時点では)AIやノーコードで初速は上がるが、仕様の曖昧さ、誤情報、セキュリティ、権利、説明責任の問題はむしろ増えている
- だからこそ、AIに任せる部分と自分で判断する部分を切り分ける力が必要になる

#### 具体的な評価基準

**知識・情報収集力（30%）**
- 開発環境、データベース、フィジカルコンピューティング、Webアプリ開発の基礎知識
- AIを活用しながら必要な情報を収集し、学習や制作に活用できるか
- AIの出力を検証し、適切な技術・手法を選択できるか
- **評価のポイント**:
  - 授業内課題で基礎知識を正しく活用できているか
  - 事前事後学習で自ら調べ、整理できているか
  - AI活用時に出力の妥当性を確認しているか

  **論理的思考力（40%）**
- フロントエンド、バックエンド、データベース、入出力処理の関係を整理できるか
- AIを活用しながら動作するアプリケーションを論理的に設計・実装できるか
- 課題の原因分析と改善ができるか
- **評価のポイント**:
  - システム全体の構造を理解し、適切に設計できているか
  - エラーやバグに対して原因を分析し、解決できているか
  - AIの提案を鵜呑みにせず、論理的に検証しているか

  **発想・構想力（30%）**
- 課題や目的に応じてアイデアを具体化できるか
- 設計・試作・検証・改善のサイクルを回せるか
- 独自性と実現性を両立した提案ができるか
- **評価のポイント**:
  - 単なる模倣ではなく、自分なりの工夫があるか
  - ユーザー視点で価値を考えられているか
  - プロトタイプを作り、フィードバックを反映しているか
